# 需求开发流程

将模糊的业务需求转化为清晰的技术方案、接口设计和开发任务。适用于 Go/Java 后端项目和 K8s 相关功能开发。

## 1. 需求澄清（Clarify）

收到需求后，先与需求方确认以下要点：

- **背景与目标**：这个需求要解决什么问题？预期效果是什么？
- **用户场景**：谁在用？什么场景下触发？频率如何？
- **成功标准**：怎么验收？PM 判断"做好了"的依据是什么？
- **约束条件**：时间节点、兼容性要求、性能要求（QPS/延迟/并发量）
- **依赖关系**：依赖哪些上游/下游系统？是否有外部依赖？

> 如果需求描述模糊，请用提问的方式引导用户补充上述信息。

## 2. 技术方案设计（Design）

确定需求清晰后，输出技术方案：

### 核心决策
- **架构变更**：是否需要新服务/新模块/新 CRD？还是现有组件扩展？
- **数据模型**：新增/修改哪些数据结构、数据库表、CRD 定义？
- **接口定义**：API（REST/gRPC）签名、消息队列 Topic、事件格式
- **状态机**：资源或业务流程的状态流转图

### 重点关注（Go + K8s 项目）
- CRD 变更的兼容性（存量资源如何处理）
- Controller 重入（Reconcile 幂等性）
- 多集群/多租户场景下的隔离性
- RBAC 权限最小化
- 现有 Finalizer 和 Webhook 的影响面

### 输出物
- 技术方案文档（markdown）
- 接口定义（proto / OpenAPI）
- CRD 变更草案（如涉及）

## 3. 任务拆解（Split）

将方案拆为可并行/串行的开发任务，每个任务遵循：

```
【任务标题】
- 涉及文件/包
- 改动要点
- 依赖的前置任务
- 预估工作量（S/M/L）
- 测试要点
```

## 4. 验收标准（Acceptance）

明确每个任务的验收条件：
- 单元测试覆盖关键路径
- 集成测试（envtest / docker-compose）
- 兼容性验证（存量数据升级）
- 性能基线（对比改动前后的 QPS 和延迟）

## Examples

- "需求：给 cceone-backend 加一个 xxx 功能" → 按 1-4 逐步推进
- "需求：新增一个 CRD 用于 xxx 资源管理" → 重点关注 CRD 兼容性和 Controller 重入
- "帮我把这个需求拆成开发任务" → 先理清需求 → 出方案 → 拆任务
